I Asked AI to Use a Screen Reader

Accessibility
Drupal
AI
Screen readers

Posted on 25 Feb 2026 - 12:06 pm

It started with a simple idea. Can I explain to stakeholders just how bad the experience of using their software is for someone using a screen reader?

I've been auditing a number of applications for compliance with WCAG 2.1 AA to help organizations understand their risks with the upcoming Department of Justice ADA Ruling looming. In some of the conversations with stakeholders, they don't always understand what the lack of support for these standards truly means for someone using an assistive device. The results were truly astounding and enlightening.

I've anonymized the prompt and the result by removing the agency name, but I left in a reference to the application name. This is not to say that all implementations of Accela are not accessible, but this particular instance clearly needs some remediation.

The prompt I gave Claude Code Sonnet 4.6 was very simple:

Using the chrome browser I just opened, can you visit the [redacted] application and provide a textual description of what a visitor using NVDA screen reader software would experience using the application. Write it as a transcript from the point of view of that user.

(Edited) Keep in mind, Claude doesn't actually have access to NVDA. It's response is to pretend it does. It's not perfect. An accessibility specialist in Drupal Slack was kind enough to provide a cursory review and called out a couple of issues with the transcript. (NVDA would not read the image filename if there is no alt text for example.) While it's not perfect, it is directionally accurate for demonstrating the experience.

I also think it is important to call out that I don't think even a detailed MCP running full screen reader tests can or should be a replacement for human testing (yet?). There is a lot of nuance and skill to using a screen reader. But for sharing with an application owner to explain the shortfalls of their software, this approach is incredibly valuable.

Here is the full transcript it gave me.

Permit Search Application — NVDA Screen Reader Experience Transcript

Environment: NVDA 2024, Chrome, Windows | Browse mode (default)
Coverage: Homepage → Search Applications (Building Permits)

Part 1: Homepage

The user opens Chrome and navigates to the application homepage.

"Accela Citizen Access"

The page title gives no indication this is a government service or what the application is for.

NVDA begins reading from the top of the page in browse mode.

"[agency-logo] dot P N G graphic"

The agency seal has no alt attribute. NVDA reads the image filename letter by letter. The user has no idea this is an agency logo.

The user presses Tab to reach the first interactive element.

"Skip to main content. Please make sure the Accessibility Support check box is marked. link"

The skip link text is disorienting. It directs the user to mark an "Accessibility Support check box" that does not exist anywhere on the page. A screen reader user expecting a standard skip link instead receives confusing instructions. The link target is # — activating it moves focus nowhere.

"home Home link"

Part 2: Search Applications

The user activates the "search Search Applications link" under Building Permits and waits for the page to load.

"Accela Citizen Access"

The page title is identical to the homepage. The user receives no confirmation they have navigated to a new page or what that page contains.

The user presses H to navigate by headings and orient themselves:

"Message Bar heading level 1"

This is a visually hidden notification container — not a useful orientation point.

"Search for Records heading level 1"

"General Search heading level 1"

The page has three H1 elements and no subordinate headings. The heading structure conveys no hierarchy and cannot help the user understand how the form is organized.

After tabbing through the repeated navigation — identical to the homepage — the user reaches the main search form with no announcement that they have arrived at it:

"Search All Records check box not checked"

The user's first interaction with the form is a checkbox with no surrounding context. No heading or group label announces that a search form has begun.

"Record Number: edit"

"Record Type: — Select — combo box"

After a short delay, NVDA reads the title attribute:

"Select a dropdown value to refresh dropdown options in associated fields or refresh the Additional Criteria Form. Keyboard Operating Instruction: Please hold ALT and press UP ARROW to expand the list, then press UP ARROW or DOWN ARROW to select an item, and finally press ENTER."

This 45-word tooltip is announced every time the user focuses this field. The embedded keyboard instruction is also inaccurate for standard NVDA and Chrome — the correct method is Alt+Down Arrow or Space, not the sequence described.

The dropdown has an onchange event handler. If the user presses a Down Arrow key to explore options and inadvertently lands on a selection, the page reloads and the user's other entries may be lost.

"Start Date: M M slash D D slash Y Y Y Y edit"

"State License Number: edit"
"Name of Business: edit"
"Business License # edit"

"Street No. From edit"

This field's accessible name comes from its title attribute only — there is no <label> element. The visible "Street No.:" label on screen is in a separate table cell and is not programmatically associated.

"Street No. To edit"

Same situation. The user hears two fields identified only as "From" and "To" with no stated relationship to each other.

"Direction: — Select — combo box"
"Keyboard Operating Instruction: Please hold ALT and press UP ARROW…"

"Search button"
"Clear button"

Summary

A screen reader user attempting to search for a building permit would need to tab past approximately 40 interactive elements — skip links that go nowhere, navigation dropdowns whose triggers are unreachable, eight module tabs with no selected state, and ten repeated "search Search Applications" links — before reaching the search form. Once there, the page title gives no confirmation they are in the right place, the heading structure offers no way to orient within the form, and the Record Type dropdown's onchange behavior risks resetting their work if they attempt to explore its options with the keyboard.

End of AI-generated transcript.

What this means for accessibility testing

This completely changes what accessibility testing looks like for applications. AI-generated transcripts are just the start. I could easily shift the prompt to provide feedback on what I need to change to achieve compliance with WCAG 2.1. Want to take it a step further? I could have it automatically create the issues in my developer backlog to address and have it take a stab at making those fixes.

Suddenly, there is no excuse for building an application that is inacessible to screen readers. And I could apply this same approach for any other accessibility accommodations.

I've updated the post to tag it with Drupal based on a conversation in Drupal Slack. While the technique above would work for any given site built with any technology, I can say that Drupal sites fair much better than most for accessibility tests because it is a focus of the community. We even have guidelines and gateways for making sure Drupal core is as accessible as possible.
